The Model Context Protocol (MCP) enables agentic workflows to connect with external services, APIs, and tools. This page provides a discovery hub for available MCP servers and shared workflow templates.

## GitHub MCP Server

The GitHub MCP server provides comprehensive access to GitHub's API for repository management, issue tracking, pull requests, and more. This is the primary tool for workflows that interact with GitHub.

<CardGrid>
  <Card title="Remote Mode (Recommended)" icon="rocket">
    Hosted GitHub MCP server with faster startup and no Docker required.

    ```yaml wrap
    tools:
      github:
        mode: remote
        toolsets: [default]
    ```
  </Card>
  <Card title="Local Mode (Docker)" icon="seti:docker">
    Run the GitHub MCP server locally in a Docker container.

    ```yaml wrap
    tools:
      github:
        mode: local
        toolsets: [default]
    ```
  </Card>
</CardGrid>

### Toolsets

Toolsets group related GitHub API operations. Using toolsets instead of individual tool names provides stability across MCP server versions.

| Toolset | Description | Example Tools |
|---------|-------------|---------------|
| `context` | User and team information | `get_me`, `get_teams`, `get_team_members` |
| `repos` | Repository operations | `get_repository`, `get_file_contents`, `search_code`, `list_commits` |
| `issues` | Issue management | `list_issues`, `create_issue`, `update_issue`, `search_issues` |
| `pull_requests` | PR operations | `list_pull_requests`, `create_pull_request`, `search_pull_requests` |
| `users` | User profiles | `get_user`, `list_users` |
| `actions` | Workflow and run management | `list_workflows`, `list_workflow_runs`, `get_workflow_run` |
| `discussions` | GitHub Discussions | `list_discussions`, `create_discussion` |
| `code_security` | Security scanning | `list_code_scanning_alerts`, `get_code_scanning_alert` |
| `labels` | Label management | `get_label`, `list_labels`, `create_label` |
| `notifications` | User notifications | `list_notifications`, `mark_notifications_read` |
| `orgs` | Organization management | `get_organization`, `list_organizations` |
| `projects` | GitHub Projects | Project board operations |
| `gists` | Gist operations | `create_gist`, `list_gists` |
| `search` | Advanced search | Search across repos, code, users |
| `dependabot` | Dependency management | Dependabot alerts and updates |
| `secret_protection` | Secret scanning | Secret detection and management |
| `security_advisories` | Security advisories | Advisory management |
| `stargazers` | Repository stars | Star-related operations |

**Default Toolsets**: `context`, `repos`, `issues`, `pull_requests`, `users`

**Common Patterns**:
- Read-only analysis: `[default]` or `[context, repos]`
- Issue and PR management: `[default, discussions]`
- CI/CD workflows: `[default, actions]`
- Security workflows: `[default, code_security, secret_protection]`
- Full access: `[all]`

See [Tools Reference](/gh-aw/reference/tools/#github-toolsets) for complete documentation.

### Using Shared Templates

Import a shared MCP template into your workflow:

```yaml wrap
---
on: workflow_dispatch
engine: copilot
imports:
  - shared/mcp/brave.md      # Import Brave Search MCP
  - shared/mcp/datadog.md    # Import Datadog MCP
---

# Research and Monitoring Agent

Search the web and check Datadog dashboards for relevant information.
```

<Aside type="tip">
Templates handle the MCP server configuration automatically. You only need to configure the required secrets in your repository settings.
</Aside>

## HTTP MCP Servers

HTTP MCP servers connect to remote APIs without running local containers.

```yaml wrap
mcp-servers:
  datadog:
    url: "https://mcp.datadoghq.com/api/unstable/mcp-server/mcp"
    headers:
      DD_API_KEY: "${{ secrets.DD_API_KEY }}"
      DD_APPLICATION_KEY: "${{ secrets.DD_APPLICATION_KEY }}"
    allowed:
      - search_datadog_dashboards
      - search_datadog_metrics
```

**Configuration Options**:
- `url`: The HTTP endpoint for the MCP server
- `headers`: HTTP headers for authentication (supports GitHub Actions secrets)
- `allowed`: List of tools to enable (use `["*"]` for all tools)

**Examples of HTTP MCP Servers**:
- Datadog monitoring and observability
- DeepWiki documentation (no auth required)
- Microsoft Learn documentation (no auth required)
- Svelte documentation (no auth required)
- Tavily AI search

## Container-Based MCP Servers

Run MCP servers in Docker containers for isolated execution.

```yaml wrap
mcp-servers:
  notion:
    container: "mcp/notion"
    env:
      NOTION_API_TOKEN: "${{ secrets.NOTION_API_TOKEN }}"
    allowed:
      - search_pages
      - get_page
      - query_database
```

**Configuration Options**:
- `container`: Docker image name (e.g., `mcp/notion`, `mcp/arxiv-mcp-server`)
- `version`: Container version tag (defaults to `latest`)
- `env`: Environment variables passed to the container
- `args`: Additional container arguments
- `entrypointArgs`: Arguments passed to the container entrypoint
- `allowed`: List of tools to enable

## Command-Based MCP Servers

Run MCP servers as local processes using npm, Python, or other package managers.

```yaml wrap
mcp-servers:
  sentry:
    command: "npx"
    args: ["@sentry/mcp-server@0.23.1"]
    env:
      SENTRY_ACCESS_TOKEN: "${{ secrets.SENTRY_ACCESS_TOKEN }}"
    allowed:
      - search_issues
      - get_issue_details
      - analyze_issue_with_seer

  markitdown:
    command: "markitdown-mcp"
    allowed: ["*"]
```

**Configuration Options**:
- `command`: The command to run (e.g., `npx`, `uvx`, `python`)
- `args`: Command arguments
- `env`: Environment variables
- `allowed`: List of tools to enable

**Common Commands**:
- `npx`: Run npm packages without installation
- `uvx`: Run Python packages using uv
- `python`: Run Python scripts directly
